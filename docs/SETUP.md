# One-time setup

Everything you need to stand up a fresh environment (or understand how the live
one is wired). The production app is already fully configured; this is the
reference for reproducing it or onboarding a new developer.

## 1. Environment files

Two gitignored files at the repo root.

`.env.local` (browser-safe, `VITE_` vars are bundled into the client):

```
VITE_SUPABASE_URL=https://<project-ref>.supabase.co
VITE_SUPABASE_ANON_KEY=<anon / publishable key>
VITE_VAPID_PUBLIC_KEY=<web push VAPID public key>
```

`.secrets.local` (server/ops only, NEVER shipped to the browser):

```
SUPABASE_ACCESS_TOKEN=<supabase CLI token>
SUPABASE_DB_PASSWORD=<database password>
SUPABASE_SERVICE_ROLE_KEY=<service role / secret key>
VAPID_PRIVATE_KEY=<web push VAPID private key>
CRON_SECRET=<shared secret the cron jobs send to the edge functions>
GMAIL_APP_PASSWORD=<16-char Gmail app password, not the account password>
```

Only the **anon** key belongs in `.env.local`; the service role key bypasses RLS
and must stay in `.secrets.local`. The project uses new-style API keys
(`sb_publishable_` / `sb_secret_`), not legacy JWTs.

## 2. Supabase project

1. supabase.com, New project. Note the Project URL and, under Settings > API, the
   anon key and service role key.
2. `supabase link` the CLI to the project (uses `SUPABASE_ACCESS_TOKEN`).
3. Apply schema: `supabase db push` (runs everything in `supabase/migrations/`).
4. Load content: apply `supabase/seed.sql` via `node scripts/db-exec.mjs supabase/seed.sql`.

## 3. Auth URLs

Authentication > URL Configuration:

- Site URL: the production URL (and `http://localhost:5173` for local dev).
- Redirect URLs: add both the production origin and `http://localhost:5173`. The
  man lands here after the Google round-trip.

## 4. Google sign-in (primary)

1. Google Cloud Console > APIs & Services > Credentials > Create OAuth client ID >
   Web application.
2. Authorized redirect URI: `https://<project-ref>.supabase.co/auth/v1/callback`
   (copy the exact value from Supabase > Authentication > Providers > Google).
3. In Supabase > Authentication > Providers > Google: enable it, paste the Client
   ID and secret.
4. Publish the OAuth consent screen (or add test users while in testing).

## 5. Email 6-digit code (fallback) + custom SMTP

- To send a 6-digit code instead of a magic link: Authentication > Email Templates
  > Magic Link, put `{{ .Token }}` in the body. The app calls `verifyOtp` with
  whatever the man types.
- Login codes and the weekly email come from the ManAlive domain via custom SMTP.
  In Supabase > Authentication > SMTP settings, use the Gmail account with an
  **app password** (16 chars, generated in Google account security, not the normal
  password). Host `smtp.gmail.com`, port 465, TLS.
- The same `GMAIL_APP_PASSWORD` must be set in THREE places and kept in sync:
  Supabase Auth SMTP, the `send-weekly-digest` edge function secret, and
  `.secrets.local`. Rotating it means updating all three.

## 6. Web Push (VAPID)

1. Generate a VAPID key pair (e.g. `npx web-push generate-vapid-keys`).
2. Public key goes in `.env.local` as `VITE_VAPID_PUBLIC_KEY`; private key in
   `.secrets.local` as `VAPID_PRIVATE_KEY` and as the `send-reminders` function
   secret.
3. iOS only allows Web Push after the PWA is added to the home screen; the
   Settings screen primes permission at the right moment.

## 7. Edge functions and cron

- Deploy: `supabase functions deploy send-reminders` and
  `supabase functions deploy send-weekly-digest`.
- Function secrets: set `CRON_SECRET`, `GMAIL_APP_PASSWORD` (digest), and the
  VAPID keys (reminders) via `supabase secrets set`.
- Scheduling: `pg_cron` calls each function hourly (migrations
  `..._schedule_reminders.sql` and `..._schedule_weekly.sql`). The "who is due
  right now" logic lives in the SQL functions `reminders_due_now()` and
  `weekly_digests_due()`, so the functions themselves stay simple.

## 8. Assets

- App icons and the notification icon are generated by `node scripts/gen-icons.mjs`
  into `public/`. Replace the source art there and re-run to refresh.

## Verify it works

- `npm run test:rls` : confirms members cannot read each other's data.
- `node scripts/send-test-email.mjs` : SMTP path.
- `node scripts/send-test-push.mjs` : push path.
- Sign in, confirm you land on Today with the right day for your start date.
